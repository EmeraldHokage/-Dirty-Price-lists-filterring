# Техническое задание
## Веб-приложение для загрузки, очистки и выгрузки CSV-прайс-листов

### 1. Общая информация
- Тип продукта: веб-приложение.
- Назначение: прием CSV-файла с «грязными» данными прайс-листа, автоматическая очистка и предоставление пользователю готового к скачиванию очищенного CSV-файла.
- Формат результата: CSV-файл, пригодный для дальнейшей загрузки в учетные системы, таблицы или внутренние базы.

### 2. Цель проекта
Создать сервис, который позволяет пользователю:
1. загрузить CSV-файл с прайс-листом;
2. автоматически обработать и очистить данные по заранее определенным правилам;
3. просмотреть результат очистки;
4. скачать очищенный CSV-файл.

### 3. Проблема, которую решает продукт
Прайс-листы часто приходят в неунифицированном виде:
- разный разделитель;
- лишние пробелы и переносы строк;
- некорректные заголовки;
- пустые строки;
- дубли;
- смешанные форматы чисел и валют;
- символы, мешающие загрузке в CRM, ERP или Excel.

Приложение должно автоматизировать первичную нормализацию таких файлов.

### 4. Область применения
Приложение используется для обработки CSV-файлов прайс-листов, содержащих:
- наименование товара;
- артикул;
- категорию;
- цену;
- остаток;
- описание;
- другие пользовательские поля, если они присутствуют в файле.

### 5. Основные сценарии использования
#### 5.1 Загрузка файла
Пользователь открывает страницу приложения и загружает CSV-файл через интерфейс.

#### 5.2 Обработка файла
Система анализирует структуру файла, определяет разделитель и применяет правила очистки.

#### 5.3 Предпросмотр результата
Пользователь видит краткую информацию:
- количество строк до и после очистки;
- количество удаленных дублей;
- количество пустых/некорректных строк;
- список основных примененных правил.

#### 5.4 Скачивание результата
Пользователь скачивает очищенный CSV-файл.

### 6. Функциональные требования
#### 6.1 Загрузка
- Система должна принимать файл формата `.csv`.
- Размер файла должен быть ограничен настраиваемым лимитом.
- При загрузке должен выполняться контроль расширения и базовой валидности файла.

#### 6.2 Определение структуры
Система должна автоматически:
- определять разделитель (`;`, `,`, `tab` или другой распространенный);
- распознавать наличие заголовка;
- определять кодировку, если это возможно;
- обрабатывать строки с разным числом колонок.

#### 6.3 Очистка данных
Система должна выполнять очистку по правилам:
- удаление пустых строк;
- удаление строк с явной структурной ошибкой, если их невозможно восстановить;
- удаление дублей;
- обрезка лишних пробелов в начале и конце значений;
- приведение множественных пробелов к одному;
- удаление служебных символов и «мусора», мешающего корректному чтению;
- нормализация кавычек и переносов строк внутри ячеек;
- унификация формата чисел;
- унификация формата цен;
- сохранение исходного смысла данных без необоснованного искажения.

#### 6.4 Работа с заголовками
- Если заголовок присутствует, система должна сохранить его и привести к единому виду.
- Если заголовка нет, система должна предложить определить его автоматически или работать без заголовка, если это допустимо.

#### 6.5 Предпросмотр
- Пользователь должен иметь возможность посмотреть результат до скачивания.
- Предпросмотр должен показывать первые строки очищенного файла.
- Пользователь должен видеть статистику обработки.

#### 6.6 Скачивание
- Очищенный файл должен скачиваться в формате `.csv`.
- Имя файла должно быть понятным и отличаться от исходного.
- При необходимости система может добавлять суффикс вроде `cleaned`.

#### 6.7 Обработка ошибок
Система должна показывать понятные сообщения, если:
- файл не является CSV;
- файл поврежден;
- файл пустой;
- структура слишком сильно нарушена;
- превышен лимит размера;
- невозможно определить формат данных.

### 7. Правила очистки данных
Ниже описаны базовые правила, которые должны быть реализованы в первой версии.

#### 7.1 Текстовые поля
- Удалять пробелы по краям.
- Убирать повторяющиеся пробелы внутри текста.
- Нормализовать переносы строк.
- Сохранять регистр, если нет отдельного правила на приведение.

#### 7.2 Числовые поля
- Приводить числа к единому формату.
- Поддерживать варианты записи с запятой и точкой.
- Удалять разделители тысяч, если они мешают корректной интерпретации.

#### 7.3 Поле цены
- Очищать от символов валюты и лишних пробелов.
- Сохранять только числовое значение и, при необходимости, отдельно знак валюты, если он распознается.

#### 7.4 Дубликаты
- Считать дубликатом строку, полностью совпадающую по набору значений.
- По возможности выявлять частичные дубликаты по ключевым полям, если это предусмотрено настройками.

#### 7.5 Пустые и некорректные строки
- Пустые строки удалять.
- Строки с невалидной структурой помечать в отчете.
- Если строку можно восстановить без потери данных, система должна пытаться это сделать.

#### 7.6 Кодировка и символы
- Приложение должно корректно работать с популярными кодировками, используемыми в CSV-файлах прайс-листов.
- Символы, не влияющие на смысл, должны быть удалены или нормализованы.

### 8. Интерфейс пользователя
Интерфейс должен быть минималистичным и состоять из:
- области загрузки файла;
- блока статуса обработки;
- блока предпросмотра результата;
- кнопки скачивания очищенного файла;
- блока ошибок и уведомлений.

### 9. Требования к результату
Очищенный CSV-файл должен:
- сохранять исходные данные, где это возможно;
- иметь корректную структуру строк и колонок;
- быть пригодным для открытия в Excel, Google Sheets и импортных системах;
- не содержать очевидного мусора, пустых строк и лишних символов;
- быть сформирован в одном и том же формате на повторных запусках при одинаковых входных данных.

### 10. Нефункциональные требования
- Приложение должно работать в браузере.
- Интерфейс должен быть адаптивным для desktop и mobile.
- Время обработки должно быть приемлемым для типового прайс-листа.
- Обработка должна быть стабильной при повторной загрузке одного и того же файла.
- Ошибки должны быть понятны пользователю без технических терминов.

### 11. Ограничения
- На первом этапе приложение работает только с CSV.
- Приложение не редактирует файлы Excel напрямую.
- Не требуется ручная правка данных на уровне отдельных ячеек, если это не предусмотрено отдельной версией.
- Не требуется хранить файлы пользователя на сервере дольше, чем необходимо для обработки, если отдельно не оговорено иное.

### 12. Критерии приемки
Проект считается выполненным, если:
1. пользователь может загрузить CSV-файл;
2. система обрабатывает файл и очищает данные;
3. пользователь видит результат очистки и краткую статистику;
4. пользователь может скачать очищенный CSV;
5. ошибки загрузки и обработки отображаются понятно;
6. результат обработки повторяем и предсказуем;
7. приложение корректно работает с типовыми «грязными» CSV-прайс-листами.

### 13. Что нужно уточнить перед разработкой
- Какие именно столбцы считаются обязательными?
- Нужно ли автоматически переименовывать заголовки?
- Какие правила считать допустимыми для удаления строки, а какие только для пометки?
- Нужно ли сохранять журнал изменений по каждой строке?
- Требуется ли поддержка пакетной обработки нескольких файлов?
- Нужно ли хранить загруженные файлы на сервере или удалять сразу после обработки?

### 14. Архитектура интеграции с OpenAI API
Очистка данных должна выполняться с использованием OpenAI API как части серверной обработки. Архитектура первой версии должна быть построена так, чтобы:
- браузер не получал доступ к API-ключу;
- ключ OpenAI хранился только в переменных окружения на стороне сервера;
- обработка CSV проходила через серверный слой, который вызывает OpenAI API;
- перед отправкой в модель выполнялась локальная проверка файла и базовая структурная нормализация;
- после ответа модели выполнялась проверка результата на валидность CSV;
- при ошибке AI-обработки система могла возвращать понятное сообщение пользователю.

#### 14.1 Роль OpenAI API в системе
OpenAI API используется для:
- анализа «грязных» строк и восстановления смысла данных;
- нормализации неоднозначных значений;
- приведения названий колонок к единому виду;
- обработки случаев, где обычных правил очистки недостаточно.

#### 14.2 Разделение ответственности
Система должна разделять обязанности между локальной логикой и OpenAI API:
- локальная логика отвечает за чтение CSV, определение разделителя, кодировки, структуры и базовую валидацию;
- OpenAI API отвечает за семантическую очистку и интеллектуальную нормализацию;
- постобработка на сервере отвечает за приведение ответа к строгому CSV-формату.

#### 14.3 Конфигурация через `.env`
Для настройки интеграции должен использоваться файл `.env`, в котором задаются секреты и параметры окружения:
- `OPENAI_API_KEY` - ключ доступа к OpenAI API;
- `OPENAI_MODEL` - имя модели, используемой для очистки;
- `OPENAI_TIMEOUT_MS` - тайм-аут запроса;
- `OPENAI_MAX_RETRIES` - число повторных попыток при сбоях сети или API.

#### 14.4 Требования к безопасности
- API-ключ нельзя хранить в клиентском коде.
- API-ключ нельзя передавать в браузер.
- Логи не должны содержать полный секретный ключ.
- При необходимости можно хранить только технические метаданные обработки без исходного содержимого чувствительных данных.

#### 14.5 Требования к стабильности
- Если OpenAI API временно недоступен, система должна сообщать об этом пользователю.
- При частичном сбое обработка не должна приводить к повреждению файла.
- Повторный запуск на том же входном файле должен давать предсказуемый результат при одинаковых настройках модели.

### 15. Итог
Данное ТЗ описывает первую версию веб-приложения для очистки CSV-прайс-листов с использованием OpenAI API. Документ можно использовать как основу для проектирования интерфейса, серверной архитектуры, логики обработки и последующей постановки задач на разработку.
